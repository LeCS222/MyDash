# Флоу: ревью → тест → коммит

Оркестрация приёмки готовой фичи. Основной агент координирует двух субагентов
по паттерну «оркестратор» и коммитит только при полном успехе.

## Последовательность

```
- [ ] 1. Код-ревью (субагент code-reviewer)
- [ ] 2. Тестирование (субагент feature-tester) — только если ревью прошло
- [ ] 3. Git-коммит — только если тест прошёл
```

Шаги строго последовательны: тестировщик запускается сразу после ревьюера, а не параллельно.

## Шаг 1. Код-ревью

Запусти субагента `code-reviewer` (foreground) через Task tool. Передай контекст:
что за фича, какие файлы изменены.

- Вердикт `CHANGES_REQUESTED` или есть 🔴 Critical / нарушения Cursor rules →
  **останови флоу**. Сообщи замечания пользователю и не переходи к тестам.
- Вердикт `APPROVED` → переходи к шагу 2.

## Шаг 2. Тестирование

Сразу после успешного ревью запусти субагента `feature-tester` (foreground)
через Task tool. Передай, какой сценарий фичи проверять.

- Вердикт `FAIL` → **останови флоу**. Сообщи проблемы пользователю. Коммита нет.
- Вердикт `PASS` → переходи к шагу 3.

## Шаг 3. Git-коммит

Только при `APPROVED` + `PASS`:

1. `git status` и `git diff` — проверь, что коммитится, и отсутствие секретов.
2. Добавь релевантные изменённые файлы (`git add`).
3. Закоммить с осмысленным сообщением, отражающим суть фичи (передавай текст
   через HEREDOC). Соблюдай Git Safety Protocol: без изменения git config, без
   `--no-verify`, без force-push; `--amend` — только по явному запросу.
4. `git status` для подтверждения успеха.

## Отчёт

В конце верни сводку по всем трём шагам:

```
Ревью:  APPROVED | CHANGES_REQUESTED (+ причины)
Тест:   PASS | FAIL (+ причины)
Коммит: <hash и сообщение> | пропущен (причина)
```

## Правила

- Не пропускай шаги и не меняй порядок: сначала ревью, затем тест, затем коммит.
- Любой неуспех выше по флоу отменяет последующие шаги. Коммит — только при полном успехе.
- Не коммить без явного разрешения пользователя запустить этот флоу.
